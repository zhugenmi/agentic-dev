# 基于 LangGraph + MCP + Skill 的多 Agent 编程助手项目设计

> 工程化方案：把“能写代码的 Agent Demo”升级为“可控、可回放、可评测、可扩展的编程助手系统”。

---

## 1. 项目背景：为什么要做这个项目？

过去两年，AI Agent 的热度持续上升，但很多“多 Agent 项目”停留在演示层：  
能聊天、能分工、能生成代码片段，却很难真正完成一个工程闭环。

对真实的软件开发来说，难点并不只是“生成代码”，而是下面这些更接近企业场景的问题：

1. **需求会变化**：用户描述往往不完整，Agent 必须先把需求规范化。
2. **代码库有上下文**：改动通常不是单文件，而是跨文件、跨模块、跨测试的联动。
3. **生成不等于可用**：代码写出来后还要通过审查、测试、修复、回归。
4. **工具调用要受控**：不能让模型随意执行命令、乱改文件、泄露密钥。
5. **系统要可评测**：面试官更关心你如何证明系统有效，而不是“看起来很智能”。

因此，这个项目不应该只是“做一个会写代码的聊天机器人”，而应当是一个：

- 能理解需求
- 能拆解任务
- 能检索代码库
- 能生成 patch
- 能审查和修复
- 能自动跑测试
- 能输出质量报告
- 能在沙箱里安全执行

的**多 Agent 编程助手系统**。

这个方向和当前企业招聘对 AI Agent 工程能力的要求是对齐的，尤其是对 **LangGraph、MCP、RAG、闭环反馈、评测与可观测性** 的要求。LangGraph 本身强调有状态图式编排、checkpoint、回放与多轮交互；MCP 则把工具、资源和提示模板标准化为可复用协议；SWE-bench 也提供了真实 GitHub issue + Docker 沙箱的评测范式。

## 2. 项目目标：做成什么样才算“有含金量”？

这个项目的目标不是做成“Agent 数量很多”，而是做成一个**工程上可解释、可运行、可评测**的系统。

### 目标一：真实软件工程闭环

系统可以接收一个自然语言任务，例如：

- “给这个项目增加 JWT 登录”
- “修复这个接口的边界条件 bug”
- “给这个模块补单元测试并提升覆盖率”

然后自动完成：

**需求理解 → 任务规划 → 代码库分析 → 代码生成 → 代码审查 → 测试生成与执行 → 修复迭代 → 最终报告**

### 目标二：突出 Agent 工程能力

重点体现以下能力：

- 基于图的工作流编排
- 基于状态的多轮循环
- 工具生态标准化接入
- 任务分层与角色隔离
- 沙箱执行与安全控制
- 结果可回放、可审计、可评测

### 目标三：能支撑面试叙述

这个项目最好能在面试中讲清楚三件事：

1. **为什么这样设计**
2. **怎么实现**
3. **怎么证明它有效**

这比单纯展示“我接了一个大模型 API”更有说服力。

## 3. 总体架构：三层控制 + 一条闭环

我的系统采用“三层控制 + 一条闭环”的架构：

- 控制层基于 LangGraph 负责状态机编排和多 Agent 调度；
- 能力层通过 MCP + Skill 抽象工具与检索能力；
- 执行层基于 Docker 沙箱保证代码运行的安全性与可复现性；
- 同时通过可观测性作为横切能力贯穿全链路；
- 最终构建了一个从规划到测试再到修复的自动化闭环。

### 3.1 控制层：LangGraph

LangGraph 负责整个系统的状态流转和编排。

它适合做什么？

- 任务拆解
- 节点调度
- 分支判断
- 循环重试
- checkpoint 持久化
- 回放调试
- 人工介入

LangGraph 的核心抽象是 **State、Node、Edge**：  
State 是共享状态；Node 是执行逻辑；Edge 决定下一步执行谁。它天生适合做“规划 → 生成 → 审查 → 修复 → 测试”的循环工作流。LangGraph 还支持 checkpoint 持久化，可以按 thread 保存每一步状态，支持 human-in-the-loop 和 time travel debugging。

**技术路线**：

1. 用 `StateGraph` 定义状态机；
2. 用 `TypedDict` / `Pydantic` 定义全局 state；
3. 用 `subgraph` 把“代码审查循环”“测试循环”拆成子图；
4. 用 `checkpoint` 保存每步状态；
5. 用 `interrupt` 支持人工确认；
6. 用 `stream` 实时输出执行轨迹。

**实现重点**：

- 所有节点输入输出都结构化；
- 每个节点只负责一个职责；
- 通过边（edge）控制路由，而不是写死 if/else；
- 把重试次数写入 state，避免死循环。

### 3.2 工具层：MCP

MCP 负责把工具能力标准化。

你可以把它理解成“Agent 的插件协议”。  不要把所有工具都直接写进 Agent 代码里，而是通过 MCP Server 暴露出来，Agent 只负责调用工具。

MCP 的标准能力包括：

- Tools：可执行函数
- Resources：可读取上下文数据
- Prompts：可复用提示模板

这让“代码库检索”“文件读取”“git 操作”“测试执行”“安全扫描”变成统一接口，而不是散落在代码里的一堆特例。MCP 官方规范明确把 tools、resources、prompts 作为服务端暴露能力。

**技术路线**：

1. 至少实现 4 个 MCP Server：
   - Git Server
   - Repo Search Server
   - Test Runner Server
   - Sandbox Executor Server
2. 每个 Server 通过标准 `tools/resources/prompts` 暴露能力；
3. Agent 不直接调用底层 shell，而是通过 MCP tool 接口调用；
4. 工具返回统一 JSON schema，方便 LangGraph 消费。

**实现重点**：

- 工具必须带 schema；
- 工具必须可审计；
- 危险操作必须加审批；
- 工具返回不能只给自然语言，最好返回结构化结果。 

### 3.3 执行层：Docker 沙箱 + Git Worktree

Agent 改代码、跑测试、装依赖都应该在隔离环境中进行。原因很简单：

- 避免误删文件
- 避免污染宿主机环境
- 避免执行危险命令
- 避免依赖冲突
- 便于回滚和复现

SWE-bench 的官方评测就是把模型生成的 patch 应用到真实仓库中，并在容器化环境里运行测试来验证是否修复成功，这个思路很适合你的项目。 

**技术路线**：

1. 每次任务创建一个独立 worktree；
2. 在 Docker 容器里安装依赖并运行测试；
3. 测试失败后保留日志和环境快照；
4. patch 通过后再合并回主分支；
5. 所有命令执行都记录到日志。

**实现重点**：

- 默认禁网；
- 最小权限；
- 资源限制；
- 可以一键销毁并重建环境。

### 3.4 记忆与检索层：短期状态 + 长期记忆

不要把所有上下文都塞进 prompt。更合理的方式是：

- **短期记忆**：当前任务的状态、阶段输出、失败日志、临时 patch
- **长期记忆**：项目知识库、常见 bug 模式、常用代码片段、历史任务记录

LangGraph 官方也明确区分了短期记忆和长期记忆：短期记忆以 thread-level checkpoint 形式存在，长期记忆用于跨会话保存用户或应用级别数据。 

**技术路线**：

1. 短期记忆放在 LangGraph state 中；
2. 长期记忆放在向量库或结构化数据库中；
3. RepoAnalyst 先做代码索引，再做相似片段检索；
4. 将检索结果按“文件级、符号级、任务级”组织。

**实现重点**：

- 不要把全部代码塞进 prompt；
- 先检索，再生成；
- 让“上下文来源”可追踪。

### 3.5 可观测性层

**技术路线**：

1. 给每次任务生成 trace id；
2. 记录每个 Agent 的输入、输出、耗时、token；
3. 记录每次测试结果和失败原因；
4. 做一个简单 dashboard 展示流程图、状态、指标。

**实现重点**：

- 能回放；
- 能定位失败点；
- 能对比不同模型路由效果。

## 4. Agent 设计：五个核心角色，足够强，也足够稳

名称固定为：

1. **Supervisor**
2. **RepoAnalyst**
3. **Implementer**
4. **Reviewer**
5. **Tester**

这五个角色已经能覆盖完整的软件工程闭环，不需要为了“看起来更复杂”而盲目增加数量。

### 4.1 Supervisor

**职责**：接收用户需求，进行任务规划和全局调度。

它负责：

- 将自然语言需求转成结构化任务
- 定义验收标准
- 拆分文件影响面
- 判断哪些子任务可以并行
- 决定是否需要人工确认
- 路由到下一个 Agent

**输出**建议包含：

- 任务目标
- 子任务列表
- 涉及文件
- 依赖关系
- 风险点
- 验收标准

Supervisor 不直接改代码，它的价值在于“把混乱需求变成可执行计划”。

### 4.2 RepoAnalyst

**职责**：分析代码库上下文。

它负责：

- 定位相关文件
- 识别项目结构
- 分析依赖关系
- 提取接口契约
- 检索历史实现
- 找到测试入口和运行方式

这个角色非常适合结合 RAG。  在大代码库场景里，模型不能靠“记忆”猜代码，只能靠检索。RepoAnalyst 的作用就是把“相关上下文”找出来，再交给后续 Agent 使用。

### 4.3 Implementer

**职责**：生成代码 patch。

它只做一件事：**按计划最小化修改代码**。  
不要把它设计成“既写代码又解释架构又补测试”的全能角色。

它的输出应该是：

- 统一 diff
- 文件级 patch
- 必要的注释
- 修改说明

Implementer 的原则是：

- 小步修改
- 保持局部一致性
- 不破坏已有结构
- 尽量以 patch 形式输出

这样更接近真实工程里的 PR 风格。

### 4.4 Reviewer

**职责**：审查代码质量、逻辑正确性、安全性和风格一致性。

Reviewer 需要关注：

- 是否违反接口约束
- 是否引入明显 bug
- 是否有安全风险
- 是否和现有代码风格冲突
- 是否存在过度设计或不必要改动

Reviewer 的输出应当是结构化的：

- `pass`
- `needs_revision`
- `reject`

并附带具体原因和修复建议。

### 4.5 Tester

**职责**：生成测试并执行测试，收集失败日志。

Tester 需要做的事情包括：

- 根据变更生成单元测试
- 执行测试用例
- 收集失败堆栈
- 判断是实现问题还是测试问题
- 把可操作反馈传给 Implementer

Tester 是闭环中最关键的角色之一。  
因为代码“看起来对”并不等于“真的对”，测试结果才是最终证据。

## 5. 工作流设计：不是死板流水线，而是受控循环图

建议将工作流设计成一个**带循环反馈的图**，而不是单向流水线。

### 5.1 标准流程

**Step 1：用户输入**

- 接收需求
- 统一需求格式
- 提取约束条件

**Step 2：Supervisor 规划**

- 任务拆解
- 风险识别
- 生成执行计划

**Step 3：RepoAnalyst 分析**

- 代码库检索
- 文件定位
- 接口与依赖分析

**Step 4：Implementer 生成 patch**

- 按计划写代码
- 尽量保持小修改

**Step 5：Reviewer 审查**

- 审查 diff
- 给出通过或修改意见

**Step 6：Tester 测试**

- 生成测试
- 执行测试
- 输出失败日志

**Step 7：修复循环**

- 如果 Reviewer 或 Tester 失败，回到 Implementer
- 重新生成 patch
- 再审查、再测试

**Step 8：Supervisor 收尾**

- 汇总结果
- 输出最终报告
- 生成 PR 文案

### 5.2 为什么要做循环

因为真实软件开发本来就是循环过程。

一个看似完整的代码 patch 可能会在下面任一环节失败：

- 编译不过
- 单测不通过
- 接口不兼容
- 风格不统一
- 安全风险高
- 和原需求偏离

所以系统必须允许“生成后回退修复”，这也是 Agent 系统与普通代码生成器的本质区别。

### 5.3 推荐的最大迭代轮数

建议设置为 **3~5 轮**。

原因：

- 轮数太少，修不完
- 轮数太多，容易陷入死循环
- 3~5 轮足够体现闭环能力

## 6. Skill 体系：把能力封装成可复用模块

Skill 是这个项目里非常值得强调的点。

### 6.1 什么是 Skill？

Skill 可以理解成“Agent 可调用的能力包”，它不等于 Agent，也不等于 Tool。

建议把 Skill 设计成四层：

- **Prompt 规范**
- **工具权限**
- **输入输出 schema**
- **风险等级 / 成本标签**

例如：

- `repo_search_skill`
- `file_read_skill`
- `patch_apply_skill`
- `test_run_skill`
- `lint_skill`
- `security_scan_skill`
- `diff_review_skill`

### 6.2 Skill 的价值

Skill 体系能带来四个明显好处：

1. **复用性**：不同 Agent 可以共享同一能力
2. **可控性**：每个 Agent 只拿到它该有的权限
3. **可观测性**：每个 Skill 都能独立统计成功率和耗时
4. **可扩展性**：新增能力时不需要重写整个系统

### 6.3 Skill 与 MCP 的关系

更合理的关系是：

- **MCP 负责对外暴露工具**
- **Skill 负责在 Agent 内部组织能力**
- **LangGraph 负责调度 Skill 所在的 Agent**

这三层不要混在一起。

## 7. MCP 工具生态：让项目像“工程系统”而不是“脚本集合”

建议最少实现 4 个 MCP Server：

### 7.1 Git MCP Server

支持：

- `status`
- `diff`
- `apply_patch`
- `checkout`
- `commit`
- `branch`

用途：  
让 Agent 以标准方式查看和修改代码变更。

### 7.2 Repo Search MCP Server

支持：

- 文件搜索
- 符号搜索
- 目录扫描
- 简单依赖分析

用途：  
帮助 RepoAnalyst 精准找到相关上下文。

### 7.3 Test Runner MCP Server

支持：

- 运行单测
- 运行 lint
- 运行类型检查
- 输出失败日志

用途：  
让 Tester 调用统一测试入口，而不是直接在 Agent 代码里写死命令。

### 7.4 Sandbox Executor MCP Server

支持：

- 在 Docker 沙箱运行命令
- 限制资源
- 记录日志
- 保护宿主机

用途：  保证整个系统的安全性和可复现性。

## 8. 应用本地模型

### 8.2 本地适合承担的任务

使用Ollama部署本地轻量coder模型，负责：

- 简单代码补全
- 小规模 patch 草稿
- 测试代码生成
- 格式整理
- 文档总结
- 低风险工具调用

### 8.1 本地模型的定位

不要让本地模型做所有事。  
更合理的做法是：

- 云端模型负责复杂推理和最终审查
- 本地模型负责快速草稿和低成本任务

这样既能体现工程思维，也能控制成本。

## 9. 模型选型：免费可用，但要把路由做成“可切换”

最合理的做法不是“每个 Agent 永远绑定一个模型”，而是做一个 **Model Router**：

- 按 Agent 角色路由；

- 按任务复杂度路由；

- 按上下文长度路由；

- 按预算与可用性路由；

- 按失败重试次数切换备选模型。 

为了让项目更稳，采用“**模型路由**”而不是“绑定单模型”。推荐策略：

- **Supervisor / Reviewer**：优先使用能力更强的云端模型
- **RepoAnalyst / Tester**：可使用中等能力模型或本地模型
- **Implementer**：优先使用代码能力强的模型
- **Fallback**：免费层模型作为备用

### 9.1 LLM选型

说明：下面的“主模型”是推荐优先级，“备选模型”用于故障切换或成本控制。  
你可以把它理解为：`primary -> fallback_1 -> fallback_2` 的路由链。

| Agent       | 主模型                                          | 备选模型                                         | 选型理由                                                 |
| ----------- | ----------------------------------------------- | ------------------------------------------------ | -------------------------------------------------------- |
| Supervisor  | **GLM-4.7**                                     | Gemini API（Free Tier）/ GLM-4.7-Flash           | 负责规划、拆解、全局调度，需要更强的推理和稳定性         |
| RepoAnalyst | **Qwen2.5-Coder-7B-Instruct（本地ollama部署）** | Qwen2.5-Coder-7B-Instruct（本地）/ GLM-4.7-Flash | 负责代码库分析、检索摘要、接口定位，优先用本地模型省成本 |
| Implementer | **GLM-4.7**                                     | Qwen2.5-Coder-7B-Instruct（本地）/ GLM-4.7-Flash | 负责生成 patch，复杂修改时优先高质量云模型               |
| Reviewer    | **GLM-4.7**                                     | Gemini API（Free Tier）/ GLM-4.7-Flash           | 审查需要更强逻辑判断和风险识别能力                       |
| Tester      | **Qwen2.5-Coder-7B-Instruct（本地ollama部署）** | GLM-4.7-Flash / Gemini Flash（Free Tier）        | 测试生成偏工具化，适合本地模型快速完成                   |

### 9.2 路由模式应该怎么设计？

建议实现一个统一的 `ModelRouter`：

```yaml
router:
  supervisor:
    primary: glm-4.7
    fallback:
      - gemini-free
      - glm-4.7-flash

  repo_analystor:
    primary: qwen2.5-coder-7B-instruct-local
    fallback:
      - qwen2.5-coder-7b-instruct-local
      - glm-4.7-flash

  implementer:
    primary: glm-4.7
    fallback:
      - qwen2.5-coder-7B-instruct-local
      - glm-4.7-flash

  reviewer:
    primary: glm-4.7
    fallback:
      - gemini-free
      - glm-4.7-flash

  tester:
    primary: qwen2.5-coder-7b-instruct-local
    fallback:
      - glm-4.7-flash
      - gemini-free
```

### 9.3 路由规则建议

路由不要只看“哪个模型更强”，还要看任务类型：

- **高推理、高风险任务**：Supervisor、Reviewer 优先 GLM-4.7；
- **代码检索、上下文整理**：RepoAnalyst 优先本地 coder 模型；
- **patch 生成**：Implementer 优先 GLM-4.7；
- **测试脚本生成、日志摘要**：Tester 优先本地模型；
- **模型不可用/超预算**：切换到免费模型或本地模型；
- **上下文超长**：优先用支持长上下文的模型。

具体方案是：

- 主力：GLM-4.7
- 本地：Qwen coder 系列：`Qwen2.5-Coder-7B-Instruct`或者`Qwen2.5-Coder-7B-Instruct`
- 免费备用：Gemini Free Tier / OpenRouter free router

## 10. 评测体系：没有评测，就没有“项目含金量”

这个项目一定要有评测，不然很容易变成“看上去很强”。

### 10.1 推荐评测维度

- 任务成功率
- 平均修复轮数
- 测试通过率
- patch 质量
- 代码覆盖率变化
- token 消耗
- 平均时延
- 安全告警次数

### 10.2 推荐基准

最适合作为主 benchmark 的是 SWE-bench 风格任务。  
SWE-bench 是基于真实 GitHub issue 的软件工程 benchmark，任务要求模型根据仓库和 issue 生成补丁；官方评测在 Docker 容器里运行仓库测试，以验证补丁是否真正修复问题。 

### 10.3 为什么这个 benchmark 适合你

因为它和你的项目目标高度一致：

- 真实代码仓库
- 真实 issue
- 跨文件修改
- 需要测试验证
- 能体现闭环能力

这比自己拍脑袋造 demo 更有说服力。

## 12. 技术亮点总结：这个项目为什么能打

如果你要把这个项目讲成“高含金量作品”，核心卖点应当是下面这几个：

### 12.1 可控性

不是一个黑盒聊天机器人，而是一个状态机式、多阶段、可回放的系统。LangGraph 的 graph/state/checkpoint 机制天然支持这一点。 

### 12.2 标准化

MCP 把工具、资源和提示模板标准化，避免工具调用散乱，方便后续扩展。 

### 12.3 真实感

SWE-bench 风格的真实仓库 + issue + Docker 测试，比玩具任务更接近真实工程。 

### 12.4 可扩展性

Skill 体系让能力可以拆分、组合、复用，不会把系统写成一坨提示词。

### 12.5 可展示性

面试时可以直接展示：

- 流程图
- 任务执行轨迹
- patch diff
- 测试日志
- 评测面板
- 成功率统计

这会比“我用了某个大模型”更有说服力。
